# OCaml 5.4.1 vs d8bb46c (5.5-beta) — fp × flambda macrobench results

**Date:** 2026-05-01
**Host:** obelisk (Intel Xeon E5-2667 v2 @ 3.30 GHz, 8 cores, governor=performance, 64 GiB RAM)
**Compilers:** `ocaml-5.4.1` (release) vs `ocaml-d8bb46c39bf5fcafb513a8ba18e667d3f8c2600a` (5.5-beta)
**Variants:** 8 — each version × {baseline, -fp, -flambda, -fp-flambda}
**Suite:** 27 macrobenchmarks, perf_grp1, runtime-events ring re-25 (32 MiB), max-domains md-2
**Invocations:** 3 per (benchmark, variant) — 216 cells total
**Config:** `running-ng/src/running/config/fp_flambda_macro_5.4.1_vs_d8bb46c.yml`
**Logs:** `running-ng/gc-sweep-logs-fp-flambda-2026-04-30/obelisk-2026-04-30-Thu-060105/`
**Companion:** `2026-05-04-fp-flambda-5.4.1-vs-d8bb46c-monolith.md` (same config, AMD Ryzen 9 9950X). The monolith run reproduces every collection-count signature exactly, so the GC-side conclusions here are confirmed; the wall-time conclusions are partially machine-specific — see §3 and the cross-machine reconciliation table below.

## TL;DR

5.5-beta changed the **major-GC pacer**, and the change is the dominant
signal across the suite:

- For workloads where 5.4 was forcing a major collection on **every**
  minor (1:1 ratio) — `owl_gc`, `zarith_pi` — the new pacer relaxes to
  2:1, dropping forced-major work in half. Net: `owl_gc` is **27-29%
  faster**, GC overhead drops 50% → 32% (and to 12% with flambda).
- For **promotion-heavy** workloads (`liq_parse_typecheck`,
  `alt_ergo_fill`) the new pacer runs **more aggressively** — more
  majors per unit allocation — and baseline regresses 7-8%. Flambda's
  better codegen reduces allocation enough that the pacer stops being
  aggressive, so the regression vanishes with flambda.
- A separate, persistent regression (~10-17%, present across all flag
  combos) hits `zarith_pi` and `ocamlc_self_compile` — both heavy
  consumers of either custom blocks or ephemerons. Likely a fast-path
  change in `caml_alloc_custom` or related.
- `dune_bootstrap`'s "flambda makes it 38% slower" is the expected
  flambda compile-time tax (more passes per file when ocamlopt.opt
  natively compiles dune itself). Not a regression.

## Findings

### 1. Major-GC pacer change — clean win on minor-saturation workloads

`owl_gc` (Bigarray finalisation, every header allocation triggers a
finaliser path that historically forced a major step) and `zarith_pi`
(custom-block storms via GMP) both showed **`minor_collections ==
major_collections`** on 5.4 — a 1:1 ratio. d8bb46c relaxes this to
**2:1** for both:

| benchmark | 5.4 (minor / major) | d8bb46c (minor / major) |
|---|---|---|
| owl_gc | 62970 / 62970 | 53693 / 26846 |
| zarith_pi | 78687 / 78687 | 106784 / 53387 |

For `owl_gc` this maps directly to wall-time savings (47% combined
with flambda, GC overhead 50% → 12%). Worth advertising as a 5.5
release win for FFI-heavy Bigarray workloads.

### 2. Pacer regression on promotion-heavy workloads

`liq_parse_typecheck` and `alt_ergo_fill` regress 7-8% on baseline +
fp because the new pacer is **more eager** when promotion is heavy:

| benchmark | 5.4 minor/major (% maj) | d8bb46c minor/major (% maj) | gc% delta |
|---|---|---|---|
| liq_parse_typecheck | 27819 / 11269 (40%) | 29168 / 14583 (50%) | 26.1% → 30.9% |
| alt_ergo_fill | 11087 / 31 (0%) | 11091 / 32 (0%) | 40.2% → 43.6% |

(The alt_ergo case has trivial major-collection counts — the gc%
movement is from `forced_major` not shown above.)

With flambda, allocation drops enough (`liq` minor: 27819 → 25341)
that the pacer stops being aggressive and the regression disappears
(`liq` flambda: 5.4=22.14s vs d8b=22.33s). This is a
**baseline-only regression** that flambda hides — but most users
don't run flambda, so it'd hit them.

### 3. Pacer × memory-subsystem interaction (was: "custom-block fast-path regression")

**Update 2026-05-04**, after running the same config on monolith
(AMD Ryzen 9 9950X) and pulling the perf-counter data: the original
"custom-block fast-path regression" hypothesis was wrong. The actual
story is the new pacer causing **3.85× more page faults**, and the
wall-time impact depends entirely on the memory subsystem.

`zarith_pi` is +16-17% slower across every flag combo on **obelisk**:

| benchmark | base | -fp | -flambda | -fp-flambda |
|---|---|---|---|---|
| zarith_pi (obelisk) | +16.6% | +17.0% | +16.6% | +16.8% |
| zarith_pi (monolith) |  +0.0% |  +3.7% |  +4.3% |  +4.8% |

Both machines exercise the *same* runtime code paths (collection counts
identical to the digit), so it's not codegen and not a runtime fast-path
change. The perf counters tell us where the wall time goes:

| metric (zarith_pi, baseline 5.4 → d8bb46c) | obelisk | monolith |
|---|---|---|
| instructions | 4.346e10 → 4.320e10 (−0.6%) | 4.229e10 → 4.145e10 (−2.0%) |
| cycles | 2.531e10 → **2.932e10** (+15.8%) | 1.422e10 → 1.427e10 (+0.4%) |
| **IPC** | 1.717 → **1.473** (−14.2%) | 2.975 → 2.904 (−2.4%) |
| page-faults | 3.84e5 → **1.48e6** (+285%) | 3.84e5 → **1.47e6** (+283%) |
| GC overhead | 35.8% → 23.1% | 31.3% → 19.9% |

The two facts that matter:

1. **Page faults grow ~3.85× on both machines under d8bb46c.** This is
   the pacer change — more aggressive minor-side reclamation churns
   working pages. It's a property of the runtime, not the host.
2. **IPC drops 14% on obelisk but only 2.4% on monolith.** Same
   instruction stream, same fault rate, different wall cost. Zen 5's
   wider OoO window, larger TLBs, and DDR5 memory subsystem absorb
   the extra TLB walks and page-fault handling; Sandy/Ivy Bridge-EP
   on DDR3 doesn't.

So the regression is real on obelisk and reproduces deterministically
across runs, but the **root cause is in the pacer's memory-access
pattern, not in `caml_alloc_custom` or custom_operations dispatch**.
This matters for what you do about it: a runtime fast-path bisect would
chase the wrong commit. A more useful direction is to look at whether
the new pacer can be tuned to coalesce the page-fault storm
(e.g. larger reclamation chunks, fewer madvise calls).

`ocamlc_self_compile` shows a similar but milder pattern: +8-10% on
obelisk, +5.6-8.2% on monolith. Same shape (collection counts agree
across machines, pacer change increases minor count, GC overhead
drops slightly), so likely the same mechanism.

### 4. Smaller per-benchmark version regressions

These regress 5-7% on baseline only (often absorbed by flambda):

- `cpdf_merge`, `irmin_mem_rw`, `menhir_sysver` — all have ≥30% GC
  overhead. Same shape as the pacer story.

### 5. Flambda compile-time tax on `dune_bootstrap`

`dune_bootstrap` is 38% slower with flambda regardless of version:
55s → 76s on 5.4, 59s → 82s on d8bb46c. This is **expected**: flambda's
ocamlopt.opt runs more optimization passes per file, and the
benchmark measures the time to compile dune itself. Not a runtime
regression; just an artifact of measuring compile-time on a
compile-time-flag.

### 6. I/O-bound: `coqc_corelib_stress`

Identical 53.8s across all 8 variants (94% GC overhead). Compiles
.v files; bottlenecked on Coq's typer + I/O, not OCaml runtime
performance. Acts as a useful "control" benchmark.

## Methodology notes

- **Median over mean** for the headline numbers (3 invocations is
  too few for CLT). Spread (max − min)/median is reported in the
  per-cell verification I ran; almost all cells are <1%, with the
  worst at 3.0% (cpdf_blacktext on 5.4-flambda, zarith_pi on
  5.4-flambda).
- **No statistical significance testing** done — that needs N ≥ ~30
  per cell, see Le Boudec 2010 / Chen 2012. Treat single-flag-combo
  deltas <5% as noise.
- **olly captures GC-internal stats** from runtime_events; perf
  captures hardware counters. The two are independent and were
  cross-checked (`event-runtime` matches wall_time within 1% on
  every cell).
- **`dune_bootstrap`** has gc_overhead=0% because olly attaches to a
  short-lived ocamlc subprocess, not the whole bootstrap. Wall time
  is the full bash wrapper (perf-captured). The pacer/codegen
  conclusions don't apply to it directly.

## Suggested follow-ups

1. **Multi-invocation re-run** with N≥10 for the four headline
   regressions (zarith_pi, ocamlc_self_compile, liq_parse_typecheck,
   alt_ergo_fill) to nail down significance.
2. **~~Bisect the d8bb46c custom-block fast-path slowdown~~** — withdrawn.
   Cross-machine perf data (see §3) shows this is not a fast-path
   change but a pacer-driven page-fault increase whose wall cost is
   memory-subsystem-dependent. A bisect would chase the wrong commit.
   Replace with: **profile the page-fault path** under d8bb46c on
   obelisk (perf record on `page-faults` event, or `strace -c` on
   madvise/mmap) to identify which pacer call site generates the
   3.85× fault increase, and whether the chunk size or madvise hint
   can be tuned.
3. **GC pacer** — coordinate with the 5.5 release notes on whether
   the promotion-heavy regression is intentional (e.g., new pacer
   tuning that prioritises reduced peak heap over wall time). The
   monolith data shows the trade-off clearly: −10 to −40% RSS on
   heap-bound workloads, in exchange for +0 to +10% wall on most.
4. **Re-test on a modern Intel host** (Sapphire Rapids+) to
   disentangle "Intel µarch" from "older-generation memory subsystem".
   The page-fault count is identical across machines, so the question
   is purely about how each memory subsystem absorbs that fault rate.
