# OCaml 5.4.1 vs 5.5-beta (d8bb46c) — fp × flambda macrobench summary

Running [macro benchmarks](https://github.com/udesou/macro-benches) suite with 27 benchmarks on obelisk, and 28 on monolith (`liq_video_frames` is new) against both 5.4.1 and the 5.5-beta commit `d8bb46c`. Eight variants per benchmark (each version × {baseline, -fp, -flambda, -fp-flambda}), two machines: obelisk (Intel Xeon E5-2667 v2, DDR3) and monolith (AMD Ryzen 9 9950X, DDR5). The more detailed results are [here](https://github.com/udesou/macro-benches/blob/master/results/2026-05-04-fp-flambda-5.4.1-vs-d8bb46c-monolith.md) and [here](https://github.com/udesou/macro-benches/blob/master/results/2026-05-01-fp-flambda-5.4.1-vs-d8bb46c-obelisk.md).

## Key findings

- The dominant signal across the whole suite is the new major-GC pacer in 5.5-beta. It looks like the GC works a bit more aggressively trading a lower max RSS for a bit of wall time in some cases. For instance, for max RSS in 5.5, we found that `cpdf_*` had -25 to -39%, `irmin_mem_rw` -28%, `htmlstream` -32%, `menhir_sysver` -20%. Wall is a small regression in most of those (`cpdf_*` +3-5%, `irmin_mem_rw` +3%, `menhir_sysver` +6%), but `devkit_htmlstream` is essentially flat (-1%), so that one ends up being a clean RSS win (-31.7%) without paying for it on wall.

- [**`owl_gc`**](https://github.com/udesou/macro-benches#owl_gc--bigarray-gromov-wasserstein-distances): 5.4 was forcing a major collection on every minor (1:1 ratio); the new pacer seems to relax that to 2:1, so forced-major work is cut in half. `owl_gc` ends up **27% faster on obelisk and 37% faster on monolith** at baseline, GC overhead drops from ~64% to ~44%, and down to 14% with flambda. Same shape on both machines. This benchmark seems to be related to off-heap memory + bigarray finalisation.

- [**`liq_video_frames`**](https://github.com/udesou/macro-benches#liq_video_frames--synthetic-large-bigarray-streaming-heap-growth-probe): this is a synthetic benchmark created from https://github.com/ocaml/ocaml/issues/14533. It seems to show a wall regression as it is (+19.5% wall, -14% rss), but we intend to do a sweep varying a few of the GC parameters (`s`, `o` and `m`) to determine whether this is a problem with the default parameters or an actual regression as pointed out in that issue. 

- It looks like promotion-heavy workloads regress. For instance, `liq_parse_typecheck` showed +8-11% wall time in the baseline runtime, with `alt_ergo_fill` showing +7-9%, on both machines. It seems like the pacer runs more aggressively here (more majors per unit allocation), although flambda seems to help improve performance. Worth doing a parameter sweep for those as well.

- `zarith_pi` looked like a +17% regression on obelisk across all flag combos. Pulling perf data from both machines: page-fault count grows ~3.85× *identically* on both. What differs is how each memory subsystem absorbs the fault rate: Zen 5's wider OoO + DDR5 + bigger TLBs make it close to free; older Sandy/Ivy Bridge-EP on DDR3 doesn't. There's also a question here whether the new machine may be benefiting from better support for prefetching in comparison to the old one.

- Smaller persistent ~5% regressions on `cpdf_merge`, `cpdf_squeeze`, `menhir_sysver`, `ocamlc_self_compile`. These are all heavy GC% workloads with shrinking heaps under 5.5, which seems to fit the "more aggressive pacer" story. RSS-wise on monolith: `cpdf_merge` -27%, `cpdf_squeeze` -26%, `menhir_sysver` -20%, but `ocamlc_self_compile` only -5%. So the trade is clear for the first three, less so for `ocamlc_self_compile` (which has the biggest wall regression of the four at +8%, with little RSS to show for it).

## What next

1. Running a sweep varying GC parameters (`s`, `o` and `m`) to find optimal parameters for regressing benchmarks and find if regressions are sensitive to those parameters.
2. Running a longer (10 iteration) benchmark to verify if results remain consistent.
3. Potentially discuss using other machines to run benchmarks so we have a clearer picture on how the machine / arch could be influencing the results.
